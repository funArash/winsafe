Available on crate feature kernel only.
Expand description

This trait is enabled with the kernel feature, and provides methods for HFILE.

Prefer importing this trait through the prelude:

use winsafe::prelude::*;

fn CreateFile(
    file_name: &str,
    desired_access: GENERIC,
    share_mode: FILE_SHARE,
    security_attrs: Option<&mut SECURITY_ATTRIBUTES<'_>>,
    creation_disposition: DISPOSITION,
    flags_and_attrs: FILE_ATTRIBUTE,
    hfile_template: Option<&HFILE>
) -> SysResult<(HandleGuard<HFILE>, ERROR)>

CreateFile static method.

The error code is also returned because it can carry information even if the file is successfully open.

Examples

Opening an existing file as read-only:

use winsafe::prelude::*;
use winsafe::{co, HFILE};

let (hfile, status) = HFILE::CreateFile(
    "C:\\Temp\\test.txt",
    co::GENERIC::READ,
    co::FILE_SHARE::READ,
    None,
    co::DISPOSITION::OPEN_EXISTING,
    co::FILE_ATTRIBUTE::NORMAL,
    None,
)?;

Opening a file for read and write. If the file doesn’t exist, create it:

use winsafe::prelude::*;
use winsafe::{co, HFILE};

let (hfile, status) = HFILE::CreateFile(
    "C:\\Temp\\test.txt",
    co::GENERIC::READ | co::GENERIC::WRITE,
    co::FILE_SHARE::NoValue,
    None,
    co::DISPOSITION::OPEN_ALWAYS,
    co::FILE_ATTRIBUTE::NORMAL,
    None,
)?;

fn LockFile(
    &self,
    offset: u64,
    num_bytes_to_lock: u64
) -> SysResult<HfileLockGuard<'_, Self>>

LockFile method.

Note that the guard returned by this method must be kept alive while the lock is used. This is necessary because, when the guard goes out of scope, it will automatically call UnlockFile.

In the example below, the guard is kept alive:

use winsafe::prelude::*;
use winsafe::HFILE;

let hfile: HFILE; // initialized somewhere

let total_size = hfile.GetFileSizeEx()?;

let _lock = hfile.LockFile(0, total_size as _)?; // keep the returned guard alive

// file read/write operations...
